# X100 — Production Roadmap

**Updated:** 2026-02-25
**Goal:** Turn the current frontend+backend into a fully working production SaaS

---

## Current State

| Layer | Status | Detail |
|-------|--------|--------|
| **Frontend** | 90% UI done | Dashboard, wizard chat, kanban, landing — all built. Most data is **mock** (INITIAL_AGENTS, static channels/knowledge/conversations) |

## BLOCK 4: Wizard → Real Agent Config (Agno pipeline)

The chat wizard collects: role, task, business description, goals, tone, documents, channels.
Currently the Platform Agent responds conversationally but **does not persist the config**.

### 4.1 Platform Agent output schema
- [ ] **`backend/agents/platform.py`** — Define Pydantic `SalesAgentConfig` output model:
  ```python
  class SalesAgentConfig(BaseModel):
      agent_name: str
      role: str              # "Sales Agent", "Support Agent", etc.
      task: str              # Main task description
      company_name: str
      product_description: str
      target_audience: str
      sales_goal: str        # Combined from goals checklist
      tone_of_voice: str     # Combined from tone checklist
      qualification_questions: list[str]
      channels: list[str]    # ["whatsapp", "telegram", ...]
      instructions: str      # Full system prompt generated by agent
  ```
- [ ] Platform Agent's final step (`wizardStep === 7`) should trigger a **structured output** call that returns this JSON

### 4.2 Auto-create agent after wizard
- [ ] **`backend/api/platform_chat.py`** — After Platform Agent returns structured config:
  1. `INSERT INTO agents` with tenant_id + config JSON
  2. For each selected channel → `INSERT INTO agent_channels`
  3. Return `agent_id` to frontend
- [ ] **Frontend `ChatView`** — On "Deploy Agent" click:
  1. Call `POST /api/platform/chat` with `action: "deploy"`
  2. Receive `agent_id` back
  3. Redirect to agent detail view or show success

### 4.3 Knowledge base upload during wizard
- [ ] **`backend/api/agents.py`** — Add `POST /api/agents/{id}/knowledge` endpoint:
  1. Accept multipart file upload (PDF, DOCX, TXT, CSV)
  2. Parse file → chunk text → generate embeddings (OpenAI)
  3. Store in `knowledge_sources` + `embeddings` tables with tenant_id
- [ ] **Frontend wizard step 5** — On file upload, actually POST to this endpoint (currently files are selected but never sent)
- [ ] After agent creation, link uploaded documents to the new agent_id

### 4.4 Agent instructions generation
- [ ] Platform Agent should generate a full system prompt (`instructions` field) based on collected config
- [ ] This prompt is what the Sales Agent uses when handling real customer conversations
- [ ] Store in `agents.config` JSONB column

**Files:** `backend/agents/platform.py`, `backend/api/platform_chat.py`, `backend/api/agents.py`, `components/Dashboard.tsx` (ChatView)

## BLOCK 5: Created Agents Actually Work

This is the core of the product — agents created by clients must handle real customer conversations.

### 5.1 Sales Agent invocation pipeline
Currently `POST /api/agents/invoke` works, but it's only called manually. Need full pipeline:

- [ ] **Incoming message flow:**
  ```
  Webhook (WhatsApp/Telegram/Instagram/Email)
    → backend/api/webhooks.py (normalize message)
    → backend/services/message_queue.py (push to Redis queue)
    → backend/workers/message_worker.py (ARQ pulls from queue)
    → Load agent config from DB by channel_id
    → create_sales_agent_full(config, tenant_id, customer_id, session_id)
    → agent.run(message)
    → Save response to DB (messages table)
    → backend/services/channel_router.py (send reply to same channel)
  ```

### 5.2 Deploy ARQ worker
- [ ] **`backend/workers/message_worker.py`** — Already written, needs:
  1. ARQ server configuration (`WorkerSettings`)
  2. Redis connection (UPSTASH_REDIS_REST_URL)
  3. Startup script / process manager
- [ ] Add `scripts/start_worker.sh` to run worker alongside FastAPI
- [ ] Handle failures: dead letter queue, retry with backoff

### 5.3 Conversation persistence
- [ ] Every message (incoming + outgoing) stored in `messages` table
- [ ] Conversations linked to `agent_id` + `customer_id`
- [ ] **Frontend Conversations kanban** — Replace mock data with `GET /api/agents/{id}/conversations`
- [ ] **Conversation detail panel** — Load real messages from `GET /api/conversations/{id}/messages`

### 5.4 Customer data tracking
- [ ] `backend/services/customer_data.py` — Already has `create_or_update_customer()`
- [ ] Sales Agent should call this tool during conversation (extract name, email, phone)
- [ ] Qualification status updates (new → qualified → converted → lost)
- [ ] **Frontend** — Show real customer data in Conversations view

### 5.5 Multi-agent per tenant
- [ ] Each tenant can have multiple agents (Sales, Support, Receptionist)
- [ ] Each agent has its own config, channels, knowledge base, conversations
- [ ] Routing: incoming message → find agent by channel_id → invoke correct agent

**Files:** `backend/workers/message_worker.py`, `backend/api/webhooks.py`, `backend/services/channel_router.py`, `backend/services/message_queue.py`, `backend/api/agents.py`

## BLOCK 6: Composio — Real Tool Integration Per Client

### 6.1 OAuth connection flow (end-to-end)
- [ ] **Frontend Settings/Agent detail** — "Connect Gmail" button → calls `POST /api/integrations/connect`
- [ ] Backend generates OAuth URL via Composio → redirect user
- [ ] User authorizes → callback → token stored in Composio per entity (entity = tenant)
- [ ] **Frontend** shows connected status with green badge

### 6.2 Per-tenant tool isolation
- [ ] Composio entity = `tenant_{tenant_id}` (already implemented in `composio_service.py`)
- [ ] Each tenant's agent uses their own OAuth tokens
- [ ] When Sales Agent invokes `GMAIL_SEND_EMAIL`, it sends from the **client's** Gmail, not ours

### 6.3 Agent tools configuration
- [ ] During wizard, collect which integrations the agent needs
- [ ] Store in `agents.config.tools` array: `["gmail", "googlecalendar", "hubspot"]`
- [ ] `create_sales_agent_full()` loads only those tools via `get_tools_for_agent(actions=[...])`
- [ ] **Frontend agent detail** — Show connected tools, allow add/remove

### 6.4 Available integrations
Currently `GET /api/integrations/apps` returns hardcoded list. Make it real:
- [ ] Fetch from Composio API: `composio_client.apps.get()` (available apps)
- [ ] Cross-reference with tenant's connected accounts
- [ ] Return: `{ app, connected: bool, account_id, connected_at }`

### 6.5 Tool execution monitoring
- [ ] Log every Composio tool call: action, status, duration, error
- [ ] Store in `tool_usage` table (ClickHouse or PostgreSQL)
- [ ] **Frontend Analytics** — Show tool usage stats per agent

**Files:** `backend/services/composio_service.py`, `backend/api/integrations.py`, `backend/agents/sales_full.py`, `components/Dashboard.tsx` (Settings view)
